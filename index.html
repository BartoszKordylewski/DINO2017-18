Komenda: git commit -a -m ""

Polecenie służy do zatwierdzenia zmian w projekcie. Opcja -a dokonuje zatwierdzenia zmian podanych plików automatycznie Jak wiemy zmiany w projekcie musimy zatwierdzić. Po dodaniu do składni polecenia git commit opcji -a każdy zmieniony plik, który jest już śledzony, automatycznie trafi do poczekalni, dzięki czemu pominiesz część git add.

Komenda: git commit -a -m ""

Komenda: git diff --staged

Polecenie porównuje zmiany z poczekalni z ostatnią migawką. Polecenie to jest bardzo ważne. Używając Gita chcemy kontrolować kod. Dzięki porównaniu zaktualizowanej wersji z poprzednią mamy większą kontrolę nad kodem, a tą umożliwa nam to polecenie.

Przykład użycia: git diff --staged

Komenda: git checkout -b "branch"

Opis: Przechodzi na gałąź o nazwie branch, jeśli taka nie istnieje to tworzy nową. Polecenie tworzy nową gałąź w obecnym repozytorium o nazwie podanej w cudzysłowiu i przełącza się na nią. Komendę wykorzystujemy gdy chcemy wprowadzić pewne zmany w kodzie, a niekoniecznie pracować w obecnej gałęzi. Polecenie możemy również użyć dodając nową osobę do repozytorium, tak aby mogła ona od razu zacząć pracować na własnej gałęzi

Przykład użycia: git checkout -b "user_branch"

Komenda: git log --graph --all

Opis: Wypisuje listę poprzednich commitów. Opcja graph dodaje graficzny podgląd gałęzi. Opcja all pokazuje też inne gałęzie. Polecenie wyświetlające historię naszego repozytorium w postacji grafu Polecenia używamy gdy chcemy zobaczyć w formie tekstowej historie Committów, obejrzeć zmiany itp.

Przykład użycia: git log --graph --all

Komenda: git branch -d "branch"

Opis: Usuwa gałąź o nazwie branch.

Przykład użycia: git branch -d "user_branch"

Komenda: git fetch "remote" "branch"

Opis: Pobiera wszystkie zmiany z remote'a o nazwie remote z gałęzi branch. Polecenie usuwa gałąź o nazwie podanej w cudzysłowiu. Polecenia używamy gdy chcemy usunąć daną gałąź z repozytorium. Możemy go użyć gdy chcemy wykluczyć część kodu napisanego w danej gałęzi przez jednego z użytkowników

Przykład użycia: git branch -d "user_branch"

Komenda: git remote add "remote_name" "git url"

Polecenie służy do dodania zdalnego repozytorium, którego nazwę podajemy w pierwszym cudzysłowiu, a URL do niego w drugim. Praca przy dużym projekcie wymaga użycia zdalnego repozytorium. Komendy używamy w celu współpracy w grupie, zarządzania zdalnym projektem itp.

Przykład użycia: git remote add "remote_name" "git url"

Komenda: git fetch "remote" "branch"

Polecenie służy do uzyskiwania danych ze zdalnych repozytorium, podajemy nazwę zdalnego repozytorium oraz gałąź. Polecenia możemy użyć gdy pracujemy nad dużym projektem w kilka osób. Polecenie pozwoli nam na uzyskanie potrzebnych nam danych ze zdalnego repozytorium

Przykład użycia: git fetch "remote" "user_branch"

git config --global user.name "John Doe"
Komenda służąca do ustawienia swojej nazwy użytkownika.
Ogólnie polecenie git config służy do ustawiania niestandardowych konfiguracji systemu.
Git korzysta z zestawu plików konfiguracyjnych, które służą do zdefiniowania ustawień dla określonych poziomów systemu.
Użycie opcji git config --system powoduje czytanie lub edytowanie konfiguracji z pliku /etc/gitconfig, która stosuje się do każdego użytkownika i repozytorium w systemie.
Opcja git config --global służy do czytania lub edytowania konfiguracji z pliku ~/.gitconfig lub ~/.config/git/config, który zawiera konfigurację specyficzną dla danego użytkownika.
Opcja git config --local czyta lub edytuje konfigurację z pliku .git/config, należącego do aktualnego repozytorium, określającego ustawienia specyficzne tylko dla niego. Jest to domyślny poziom konfiguracji, w przypadku braku określenia odpowiedniej opcji w poleceniu git config.
Ustawienia każdego z poziomów nadpisują te, które są określone na niższym poziomie, zgodnie z kolejnością: system, global, local, gdzie global nadpisuje system, a local wszytkie pozostałe. Polecenie to ustawia nam nazwę użytkownika. W cudzysłowia wstawiamy nazwę użytkownika, pozostawiając je pustę resetujemy nazwę użytkownika.

Komendy Git GRUPA CF
git config --global color.ui true
Włącza wszystkie domyślne ustawienia dotyczące kolorowania wyników komend z rodziny git na terminalu. Inne możliwe ustawienia to "false", które nigdy nie będzie pokazywało w kolorze wyników działania, oraz "always", które zawsze ustawi kolory, nawet w przypadku gdy będziesz chciał zapisać wyniki do pliku lub przekazać do innej komendy.
git commit "filename"
Commituje zmianę w repozytorium tylko pliku o nazwie "filename". Jeśli zrobimy commit i znajdziemy błąd natychmiast po tym, możemy wyofać go za pomocą komendy 'git reset'.
git branch "name"
Tworzy nową gałąź o nazwie "name", ale do niej nie przechodzi, trzeba to zrobić za pomocą 'git checkout "name"'.
git branch -v
W trybie listy wyświetla skrót sha1 i temat commita w jednej linii dla każdej sekcji head, wraz z relacją do gałęzi źródłowej (jeśli istnieje)
git push "remote" "branch"
Synchronizacja z serwerem zdalnym "remote" gałęzi "branch". Inaczej wypycha gałąź branch na server remote. Ogólna postać komendy: git push [nazwa zdalnego repozytorium] [nazwa gałęzi] Przydatna gdy chcemy udostępnić dokonane przez nas zmiany członkom zespołu, którzy pracują przy tym samym projekcie.
git log "branch" --not "branch"
Wyświetla listę commitów z gałęzi "branch" z wyjątkiem gałęzi o nazwie "branch". w konsekwencji nie wyświetli nic.
git blame -C "filename"
Rozpoznaje linie w pliku "filename" skopiowane z innych plików, które były zmodyfikowane w ramach jednego commita. Przydatne przy reorganizacji struktury programu.
git bisect (start, bad, good, reset)
Wykonuje binarne szukanie przez historię commitów, aby pomóc zidentyfikować tak szybko jak się da, który commit wporwaidził błąd. Na początek uruchamiamy 'git bisect start', aby rozpocząć, a potem 'git bisect bad' aby powiedzieć systemowi że obecny commit na którym się znajdujesz jest popsuty. Następnie, wskazujemy kiedy ostatnia znana poprawna wersja była, przy użyciu 'git bisect good [poprawna_wersja]'. Jeżeli to w nim znajduje się błąd to oznaczamy go jako bad z użyciem 'git bisect bad', a jeżeli nie wystepuje to oznaczamy jako 'git bisect good' i ponownie otrzymujemy commita w którym możemy mieć tego buga. Proces powtarzamy do znalezienia commita który wprowadza bug/zmiany. Reset to powrót do stanu z przed bisekcji, czyli powrót do oryginalnego head.
gitk -all
Sama komenda gitk ma na celu wyświetlenie zmian w repozytorium lub wyznaczonych commitów. Wizualizuje graf commitów z informacjami związanymi z każdym commitem. Opcja -all pokauzuje nam w pełni nasze repozytorium, czyli wszystkie branche, tagi etc.
git branch --merged
Pokazuje branche zmergowane w branchu w którym aktualnie się znajdujemy. Odwrotną do tego opcją jest --no-merged, która pokazuje jakich branchów jeszcze nie zmergowaliśmy. Domyślnie ta komenda pokazuje tylko lokalne branche. Aby pokazywała i lokalne i zdalne, potrzeba użyć opcji -a.
git merge upstream/master
Polecenie zacznie od próby aktualizowania, jeśli jakieś zmiany zostały przeprowadzone. W przeciwnym wypadku wyświetlony zostanie komunikat, że wszystie pliki są "Up-to-date".
